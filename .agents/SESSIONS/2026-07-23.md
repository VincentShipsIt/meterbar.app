## Session 2: Stabilize Share preview while scrolling

**Status:** Implementation complete; PR CI pending

### Root cause

- The Share receipt preview derives its height from the width proposed by the dashboard's vertical `ScrollView`.
- macOS can claim or release horizontal space for the vertical scroll indicator while scrolling.
- That gutter change altered the preview's proposed width, so its fixed 16:9 aspect ratio made the entire receipt appear to zoom.

### What was done

- Kept the dashboard's native scroll indicators and the user's system scrollbar preference intact.
- Derived an explicit preview width and height from the stable detail viewport rather than the scroll view's changing content proposal.
- Reserved the legacy vertical-scroller width so "Show scroll bars: Always" cannot overlap or resize the preview.
- Added regression coverage for responsive, capped, and constrained preview geometry.

## Session 3: Unify chrome and content on one neutral surface system

**Status:** Implementation complete; PR CI pending

### Root cause

- Session 1 (#248) neutralized **Layer 2 only**: content cards moved from a teal wash to opaque `controlBackgroundColor`.
- Every **Layer 1** chrome tint was left untouched â€” `Surface.companionTint` stayed teal, and `MeterBarDetailBackground` still painted a `codexAccent`/`appAccent` `LinearGradient` over its material.
- Result: blue/teal chrome wrapped around gray cards â€” two competing surface systems in one window.
- Compounding: in dark mode `controlBackgroundColor` (~#1E1E1E) is *darker* than `.regularMaterial`, so opaque cards read as recessed slabs rather than lifted content.

### What was done

- `Surface.companionTint`: teal â†’ luminance-matched graphite, so the popover and detail-panel glass carry no product hue.
- `MeterBarDetailBackground`: deleted the accent `LinearGradient`, leaving plain `.regularMaterial`. One edit de-blues both the main window and the fixed-size Settings window (the latter uses the same background).
- `Surface.content`: opaque `controlBackgroundColor` â†’ neutral translucent white lift (0.55 light / 0.08 dark; 0.85 / 0.14 under Increase Contrast) so cards lift off the chrome instead of punching an opaque hole in it.
- Tests first: relaxed and renamed the content-surface regression test, and added `testChromeTintIsHueNeutral` â€” the invariant whose absence let this ship.
- `.agents/DESIGN.md`: documented the content surface accurately and added an explicit "never tint one layer while the other stays neutral" rule.

### Decisions

- **No `contentOpaqueFallback` reintroduced.** A flat alpha wash is not vibrancy, so a translucent fill over an opaque backdrop already satisfies Reduce Transparency; #248's token simplification stands.
- **`sidebarMenuTint` left unchanged.** It is a control/selection accent, not a surface fill; retinting it would be scope creep on a surface-color complaint.
